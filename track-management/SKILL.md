# 轨道管理

创建、管理和完成 Conductor 轨道的指南 — 轨道是通过规范、规划和实现阶段组织功能、缺陷和重构的逻辑工作单元。

## 何时使用此技能

- 创建新的功能、缺陷或重构轨道
- 撰写或审查 spec.md 文件
- 创建或更新 plan.md 文件
- 管理从创建到完成的轨道生命周期
- 理解轨道状态标记和约定
- 使用 tracks.md 注册表
- 解释或更新轨道元数据

## 轨道概念

轨道是封装完整工作单元的逻辑工作单元。每个轨道有：

- 唯一标识符
- 定义需求的规范
- 将工作分解为任务的分阶段计划
- 跟踪状态和进度的元数据

轨道为工作提供语义组织，使得：

- 清晰的范围边界
- 进度跟踪
- Git 感知操作（按轨道回滚）
- 团队协调

## 轨道类型

### feature（功能）

新功能或能力。用于：

- 新的面向用户功能
- 新 API 端点
- 新集成
- 重大增强

### bug（缺陷）

缺陷修复。用于：

- 错误行为
- 错误条件
- 性能回归
- 安全漏洞

### chore（杂务）

维护和清理。用于：

- 依赖更新
- 配置变更
- 文档更新
- 清理任务

### refactor（重构）

不改变行为的代码改进。用于：

- 代码重组
- 模式采用
- 技术债务减少
- 性能优化（相同行为，更好性能）

## 轨道 ID 格式

轨道 ID 遵循模式：`{shortname}_{YYYYMMDD}`

- **shortname**：2-4 个词的 kebab-case 描述（如 `user-auth`、`api-rate-limit`）
- **YYYYMMDD**：ISO 格式的创建日期

示例：

- `user-auth_20250115`
- `fix-login-error_20250115`
- `upgrade-deps_20250115`
- `refactor-api-client_20250115`

## 轨道生命周期

### 1. 创建（newTrack）

**定义需求**

1. 通过交互式问答收集需求
2. 确定验收标准
3. 确定范围边界
4. 识别依赖

**生成规范**

1. 创建包含结构化需求的 `spec.md`
2. 记录功能和非功能需求
3. 定义验收标准
4. 列出依赖和约束

**生成计划**

1. 创建包含分阶段任务分解的 `plan.md`
2. 将任务组织为逻辑阶段
3. 在阶段后添加验证任务
4. 估算工作量和复杂度

**注册轨道**

1. 向 `tracks.md` 注册表添加条目
2. 创建轨道目录结构
3. 生成 `metadata.json`
4. 创建轨道 `index.md`

### 2. 实现

**执行任务**

1. 从计划中选择下一个待处理任务
2. 标记任务为进行中
3. 按工作流实现（TDD）
4. 标记任务完成并附带 commit SHA

**更新状态**

1. 更新 plan.md 中的任务标记
2. 记录 commit SHA 用于可追溯性
3. 更新阶段进度
4. 更新 tracks.md 中的轨道状态

**验证进度**

1. 完成验证任务
2. 等待检查点批准
3. 记录检查点 commit

### 3. 完成

**同步文档**

1. 如添加了功能则更新 product.md
2. 如变更了依赖则更新 tech-stack.md
3. 验证所有验收标准已满足

**归档或删除**

1. 在 tracks.md 中标记轨道为已完成
2. 记录完成日期
3. 归档或保留轨道目录

## 规范（spec.md）结构

```markdown
# {轨道标题}

## 概述

简要描述此轨道完成什么以及为什么。

## 功能需求

### FR-1：{需求名称}

功能需求的描述。

- 验收：如何验证此需求已满足

### FR-2：{需求名称}

...

## 非功能需求

### NFR-1：{需求名称}

非功能需求的描述（性能、安全等）

- 目标：具体可测量的目标
- 验证：如何测试

## 验收标准

- [ ] 标准 1：具体、可测试的条件
- [ ] 标准 2：具体、可测试的条件
- [ ] 标准 3：具体、可测试的条件

## 范围

### 范围内

- 明确包含的项目
- 要实现的功能
- 要修改的组件

### 范围外

- 明确排除的项目
- 未来考虑
- 相关但独立的工作

## 依赖

### 内部

- 此轨道依赖的其他轨道或组件
- 所需的上下文产物

### 外部

- 第三方服务或 API
- 外部依赖

## 风险和缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| 风险描述 | 高/中/低 | 缓解策略 |

## 开放问题

- [ ] 需要解决的问题
- [x] 已解决的问题 - 答案
```

## 状态标记约定

在 plan.md 中使用一致的标记：

| 标记 | 含义 | 用法 |
|------|------|------|
| `[ ]` | 待处理 | 任务未开始 |
| `[~]` | 进行中 | 正在处理 |
| `[x]` | 完成 | 任务完成（包含 SHA） |
| `[-]` | 跳过 | 刻意不做 |
| `[!]` | 阻塞 | 等待依赖 |

示例：

```markdown
- [x] **任务 1.1**：设置数据库架构 `abc1234`
- [~] **任务 1.2**：实现用户模型
- [ ] **任务 1.3**：添加验证逻辑
- [!] **任务 1.4**：集成 auth 服务（阻塞：等待 API 密钥）
- [-] **任务 1.5**：旧版迁移（跳过：不需要）
```

## 轨道操作

### 创建轨道

1. 运行 `/conductor:new-track`
2. 回答交互式问题
3. 审查生成的 spec.md
4. 审查生成的 plan.md
5. 确认轨道创建

### 开始实现

1. 读取 spec.md 和 plan.md
2. 验证上下文产物是最新的
3. 标记第一个任务为 `[~]`
4. 开始 TDD 工作流

### 完成阶段

1. 确保所有阶段任务为 `[x]`
2. 完成验证任务
3. 等待检查点批准
4. 记录检查点 SHA
5. 进入下一阶段

### 完成轨道

1. 验证所有阶段完成
2. 验证所有验收标准已满足
3. 如需要更新 product.md
4. 在 tracks.md 中标记轨道完成
5. 更新 metadata.json

### 回滚轨道

1. 运行 `/conductor:revert`
2. 选择要回滚的轨道
3. 选择粒度（轨道/阶段/任务）
4. 确认回滚操作
5. 更新状态标记

## 处理轨道依赖

### 识别依赖

在轨道创建期间，识别：

- **硬依赖**：必须在此轨道开始前完成
- **软依赖**：可以并行进行但可能影响集成
- **外部依赖**：第三方服务、API 或团队决策

### 记录依赖

在 spec.md 中，列出依赖及：

- 依赖类型（硬/软/外部）
- 当前状态（可用/待处理/阻塞）
- 解决路径（需要发生什么）

### 管理阻塞轨道

当轨道被阻塞时：

1. 用 `[!]` 标记阻塞任务并注明原因
2. 更新 tracks.md 状态
3. 在 metadata.json 中记录阻塞者
4. 如需要考虑创建依赖轨道

## 轨道规模指南

### 合适规模的轨道

目标轨道：

- 1-5 天工作完成
- 2-4 个阶段
- 总共 8-20 个任务
- 交付连贯、可测试的单元

### 过大

轨道过大的信号：

- 超过 5 个阶段
- 超过 25 个任务
- 多个不相关功能
- 预计持续时间 > 1 周

解决方案：拆分为多个有清晰边界的轨道。

### 过小

轨道过小的信号：

- 单阶段仅 1-2 个任务
- 无需有意义的验证
- 可以是另一个轨道的子任务
- 不到几小时的工作

解决方案：与相关工作合并或作为现有轨道的一部分处理。

## 规范质量清单

最终确定 spec.md 前，验证：

### 需求质量

- [ ] 每个需求有清晰的验收标准
- [ ] 需求是可测试的
- [ ] 需求是独立的（可单独验证）
- [ ] 无模糊语言（"应该快" → "响应 < 200ms"）

### 范围清晰度

- [ ] 范围内项目是具体的
- [ ] 范围外项目防止范围蔓延
- [ ] 边界对实现者清晰

### 依赖已识别

- [ ] 所有内部依赖已列出
- [ ] 外部依赖有所有者/联系人
- [ ] 依赖状态是最新的

### 风险已处理

- [ ] 主要风险已识别
- [ ] 影响评估现实
- [ ] 缓解措施可操作

## 计划质量清单

开始实现前，验证 plan.md：

### 任务质量

- [ ] 任务是原子的（一个逻辑操作）
- [ ] 任务可独立验证
- [ ] 任务描述清晰
- [ ] 子任务提供有用的详情

### 阶段组织

- [ ] 阶段分组相关任务
- [ ] 每个阶段交付可测试的内容
- [ ] 每个阶段后有验证任务
- [ ] 阶段逻辑上相互构建

### 完整性

- [ ] 所有规范需求有对应任务
- [ ] 包含文档任务
- [ ] 包含测试任务
- [ ] 包含集成任务

## 常见轨道模式

### 功能轨道模式

```
阶段 1：基础
- 数据模型
- 数据库迁移
- 基本 API 结构

阶段 2：核心逻辑
- 业务逻辑实现
- 输入验证
- 错误处理

阶段 3：集成
- UI 集成
- API 文档
- 端到端测试
```

### 缺陷修复轨道模式

```
阶段 1：复现
- 编写捕获缺陷的失败测试
- 记录复现步骤

阶段 2：修复
- 实现修复
- 验证测试通过
- 检查回归

阶段 3：验证
- 手动验证
- 如需要更新文档
```

### 重构轨道模式

```
阶段 1：准备
- 添加表征测试
- 记录当前行为

阶段 2：重构
- 增量应用变更
- 全程保持绿色测试

阶段 3：清理
- 删除死代码
- 更新文档
```

## 最佳实践

1. **一个轨道，一个关注点**：保持轨道聚焦于单一逻辑变更
2. **小阶段**：将工作分解为最多 3-5 个任务的阶段
3. **阶段后验证**：始终包含验证任务
4. **立即更新标记**：工作时标记任务状态
5. **记录 SHA**：始终记录已完成任务的 commit SHA
6. **规划前审查规范**：创建计划前确保规范完整
7. **链接依赖**：明确注明轨道依赖
8. **归档而非删除**：保留已完成轨道以供参考
9. **适当规模**：保持轨道在 1-5 天工作量
10. **清晰的验收标准**：每个需求必须是可测试的
