# 内容哲学

## 阶段 2.5（简报）和阶段 3（编写模块）的指导。每个规则都旨在保持课程视觉化、引人入胜且对非技术学习者有用。

## 视觉优先方法

- 每个屏幕的文本块限制在 **2-3 句话**
- **每个屏幕必须至少 50% 是视觉内容**——图表、卡片、动画、代码块
- 3 项以上的列表应变成卡片、流程图或步骤卡片
- 序列应变成编号步骤卡片或流动画
- 比较应变成并排列
- 使用大量留白——让内容有呼吸空间
- 每个模块至少一个"主视觉"（大图表、动画或互动元素）

## 代码与英文对照翻译

- 每个代码片段都需要并排的通俗英文解释
- 代码必须**永远不产生水平滚动条**——使用 `white-space: pre-wrap`
- 原始代码必须**完全原样呈现**，不修改或不修剪
- 选择自然简洁的片段（5-10 行）来演示概念
- 英文翻译应解释**为什么**，而不仅仅是**是什么**
- 使用会话语言："Include our API key so the server knows who we are"（包含我们的 API 密钥，这样服务器就知道我们是谁）而不是 "Set the Authorization header"（设置 Authorization 头）

## 每个屏幕一个概念

每个屏幕教一个想法。不要塞内容，而是添加更多屏幕。如果一个屏幕涵盖两个不同的概念，就把它分开。

## 隐喻策略

新概念在用代码巩固之前，先用一个现实世界的隐喻。

**严格规则：**
- 绝不能使用"餐厅"或"厨房"隐喻——它们被用得太多了
- 绝不能在跨模块重复使用相同的隐喻
- 每个模块都需要自己的隐喻，感觉自然适合那个概念
- 最好的隐喻让人觉得*对于这个概念是不可避免的*，而不是牵强的
- 如果你发现自己两次使用同一个隐喻，停下来找一个新隐喻

## 通过追踪学习

内容应该通过系统端到端地遵循真实用户操作，利用学习者对应用的现有体验。从"你点击了这个按钮"开始，追踪代码中发生了什么。

## 术语提示

每个技术术语在每个模块的首次使用时都要加虚线下划线提示。

**规则：**
- 使用 `cursor: pointer`（不是 `cursor: help`——pointer 感觉可点击且吸引人）
- 将提示附加到 `document.body` 并使用 `position: fixed`（防止被 `overflow: hidden` 容器裁剪）
- 覆盖非技术人员可能不知道的任何内容——包括软件名称、缩写语和编程概念
- 用日常语言将定义保持在一两句话
- 在定义中使用隐喻当它有帮助时
- 不要在同一屏幕内对同一术语标记两次

## 测验设计

测验出现在模块末尾（3-5 个问题），优先考虑**实际应用而非记忆**。

**问题类型层次（从高到低）：**
1. 基于场景的问题解决（"如果……你会怎么做……"）
2. 追踪练习（"如果用户点击 X，接下来会发生什么？"）
3. 模式识别（"哪种方法对这种情况更好？"）
4. 概念应用（"你会如何向 AI 解释这个？"）

**规则：**
- 错误答案应该给出**鼓励性的、非评判性的解释**
- 明确避免评分指标——"测验是一个思考练习，不是考试"
- 永远不要问"API 代表什么？"或其他定义回忆问题
- 每个问题都应该呈现一个新的场景，要求学习者*应用*他们所学到的
