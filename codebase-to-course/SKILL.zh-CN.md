# Codebase-to-Course（代码库转课程）

将任何代码库转换为一门精美、互动的课程。输出是一个**目录**，包含预构建的 `styles.css`、`main.js`、每个模块的 HTML 文件，以及组装好的 `index.html`——可直接在浏览器中打开，无需任何配置（唯一外部依赖：Google Fonts CDN）。课程通过基于滚动导航的模块、动画可视化、嵌入式测验和通俗易懂的代码英文对照来教授代码如何工作。

## 首次运行欢迎语

当技能首次触发且用户尚未指定代码库时，介绍自己并解释职能：

> **我可以把任何代码库变成一门教你它如何工作的互动课程——不需要任何编程知识。**
>
> 只需告诉我项目位置：
> - **本地文件夹** — 例如："把这个 ./my-project 变成一门课程"
> - **GitHub 链接** — 例如："从 https://github.com/user/repo 制作一门课程"
> - **当前项目** — 如果你已在代码库中，只需说"把它变成一门课程"
>
> 我会阅读代码，弄清各部分如何协同工作，然后生成一个精美的单页 HTML 课程，包含动画图表、通俗易懂的代码解释和互动测验。整个系统在浏览器中运行——无需任何配置。

如果用户提供 GitHub 链接，先克隆仓库（`git clone <url> /tmp/<repo-name>`），再开始分析。如果用户说"这个代码库"或类似表述，使用当前工作目录。

## 这门课程面向谁

目标学习者是**"感觉编程者"（vibe coder）**——通过自然语言指挥 AI 编程工具来构建软件的人，没有传统计算机科学教育背景。他们可能自己构建了这个项目（没有看过代码），或者在 GitHub 上发现了一个有趣的开源项目，想了解它的构建方式。无论哪种情况，他们都不理解底层发生了什么。

**假设学习者没有任何技术背景。** 每个计算机科学概念——从变量到 API 再到数据库——都需要用通俗易懂的语言解释，就像他们从未接触过这些概念一样。不能出现未经定义的术语，不能说"你可能知道"。语气应该像聪明的朋友在解释，而不是教授在讲课。

**他们的目标是实用的，不是学术的：**
- 具备足够的技术知识来有效地**指挥 AI 编程工具**——做出更好的架构和技术栈决策
- **检测 AI 何时出错**——发现幻觉、捕捉错误模式、知道何时不对劲
- **在 AI 卡住时进行干预**——跳出 bug 循环、调试问题、解除阻塞
- 构建更具**生产级质量和可靠性**的高级软件
- **达到足以与工程师自信讨论技术决策的技术流利度**
- **获得软件的词汇**——学习精确的技术术语，以便能够清晰明确地向 AI 编程代理描述需求（例如，知道说"命名空间包"而不是"共享文件夹那个东西"）

**他们不是在努力成为软件工程师。** 他们想要编程作为超级能力来放大他们已有的能力。他们不需要从头写代码——他们需要的是*阅读*它、*理解*它和*指挥*它。

## 为什么这种方法有效

本技能颠覆了传统的计算机科学教育。旧模式是：多年记忆概念 → 最终构建东西 → 才看到意义（大多数人在第三步之前就放弃了）。本模式是：**先构建东西 → 体验它如何工作 → 现在理解它如何工作。**

学习者拥有传统学生没有的上下文——他们*用过*这个应用，知道它做什么，他们甚至可能用自然语言描述过它的功能。课程在他们所在的地方与他们会面："你知道你点击的那个按钮吗？当你点击它时，下面是这样工作的。"

每个模块回答**"为什么我应该关心？"** 优先于"它如何工作？""为什么我应该关心？"的答案始终是实用的：*因为这些知识帮助你更好地指挥 AI、更快地调试或做出更明智的架构决策。*

基于目录的输出是刻意设计的：将 CSS/JS 与内容分离意味着 AI 永远不会重新生成样板文件，每个模块独立编写（保持输出小且高质量），组装好的 `index.html` 可以离线零配置运行。

## 流程

### 阶段 1：代码库分析

在编写课程 HTML 之前，深入理解代码库。阅读所有关键文件，追踪数据流，识别"角色阵容"（主要组件/模块），并绘制它们如何通信的地图。这里的彻底性是有回报的——你理解得越深入，课程就越好。

**需要提取的内容：**
- 主要"角色"（组件、服务、模块）及其职责
- 主要用户旅程（当某人端到端使用应用时会发生什么）
- 关键 API、数据流和通信模式
- 巧妙的工程模式（缓存、延迟加载、错误处理等）
- 真正的 bug 或陷阱（如果在 git 历史或注释中可见）
- 技术栈及选择每个部分的原因

**自己要弄清楚应用做什么**——通过阅读 README、入口文件和 UI 代码。不要要求用户解释产品——他们也可能不熟悉它。课程开头应该用通俗易懂的语言解释应用做什么（简要说明"这个东西做什么以及为什么有趣"），然后再深入其工作原理。第一个模块应该从一个具体的用户操作开始——"假设你粘贴了一个 YouTube 链接并点击分析——下面是点击时底层发生的事。"

### 阶段 2：课程设计

将课程构建为 **4-6 个模块**。大多数课程需要 4-6 个。只有当代码库确实有那么多值得讲授的不同概念时，才使用 7-8 个。少而精的模块比多而薄的模块更好。

课程弧线始终从学习者已经知道的（用户面向的行为）走向他们不知道的（底层代码）。可以把它想象成放大：先从体验开始，然后逐步剥开层。

| 模块位置 | 目的 | 为什么对感觉编程者重要 |
|---|---|---|
| 1 | "这是这个应用做什么的——以及你使用它时会发生什么" | 从产品开始（它做什么，为什么有趣），然后追踪一个核心用户操作进入代码。用具体的东西为一切奠定基础。 |
| 2 | 认识角色 | 知道存在哪些组件，这样你可以告诉 AI"把这个逻辑放在 X，而不是 Y" |
| 3 | 各部分如何对话 | 理解数据流，这样你可以调试"它没有显示出来"的问题 |
| 4 | 外部世界（API、数据库） | 知道什么是外部的，这样你可以评估成本、速率限制和失败模式 |
| 5 | 巧妙的技巧 | 学习模式（缓存、分块、错误处理），这样你可以向 AI 请求它们 |
| 6 | 当东西坏了时 | 建立调试直觉，这样你可以跳出 AI bug 循环 |
| 7 | 全貌 | 查看完整架构，这样你可以更好地决定下一步构建什么 |

这是一个**菜单，不是检查清单**。选择服务于代码库的模块——一个简单的 CLI 工具需要 4 个，不是 7 个。根据代码库的复杂性调整弧线。

**关键原则：** 每个模块都应该连接回一项实用技能——指挥 AI、调试、做决策。如果一个模块不能帮助学习者更好地*做*某事，就删除它或重新构建它，直到它能做到。

**每个模块应该包含：**
- 3-6 个屏幕（模块内流畅的子部分）
- 至少一个代码与英文对照翻译
- 至少一个互动元素（测验、可视化或动画）
- 一两个带有通用计算机科学见解的"啊哈！"提示框
- 一个将技术概念与日常生活联系起来的隐喻——但绝不能跨模块重复使用相同的隐喻，绝不能默认使用"餐厅"隐喻（它被用得太多了）。选择有机地适合特定概念的隐喻。最好的隐喻让人觉得*对于这个概念是不可避免的*，而不是牵强的。

**强制性互动元素（每个课程必须包含所有这些）：**
- **群聊动画** — 整个课程至少一个。这些是组件之间对话的 iMessage/微信风格。它们是最吸引人的元素之一，必须始终出现，即使你必须创造性地将一个模块的概念框架为角色之间的对话。
- **消息流/数据流动画** — 整个课程至少一个。角色之间逐步的数据包动画。如果代码库有任何请求/响应、数据管道或多步骤流程，都要动画化。每个代码库都有数据在某个地方流动——找到它。
- **代码 ↔ 英文对照块** — 每个模块至少一个（上述已要求，但重申：这是不可妥协的）。
- **测验** — 每个模块至少一个（选择题、场景题、拖放匹配或找 bug——任何测验类型都算）。
- **术语表提示** — 每个模块的每个技术术语首次使用时都有。

这五种元素类型是每个课程的支柱。其他互动元素（架构图、层切换、模式卡等）是可选的，应该在合适时添加。但以上五种必须始终存在——没有例外。

**不要呈现课程以供批准——直接构建。** 用户想要课程，不是计划文档。在内部设计课程，然后直接开始构建。如果他们想要更改，看到结果后会告诉你。

**设计完课程后，决定使用哪种构建路径：**

- **简单代码库**（单一用途 CLI、小型 Web 应用、库、一个清晰的入口点、5 个或更少模块）→ 直接进入阶段 3 顺序路径。
- **复杂代码库**（全栈应用、多个服务、内容重的站点、monorepo、或 6+ 个模块）→ 先进入阶段 2.5，然后阶段 3 并行路径。

### 阶段 2.5：模块简报（仅限复杂代码库）

对于复杂的代码库，在编写任何 HTML 之前，为每个模块写一份简报。这是实现并行编写的关键步骤——每份简报给代理所需的一切，无需重新阅读代码库。

阅读 `references/module-brief-template.md` 了解模板结构。阅读 `references/content-philosophy.md` 了解应该指导简报编写的内容规则。

**对于每个模块，将简报写入 `course-name/briefs/0N-slug.md`，包含：**
- 教学弧线（隐喻、开场钩子、关键见解）
- 预提取的代码片段（从代码库复制粘贴，附文件路径和行号）
- 互动元素检查清单，包含足够构建它们的细节
- 写作代理需要阅读的参考文件的哪些部分
- 前后模块涵盖的内容（用于过渡）

代码片段是节省 token 的关键步骤。通过将它们预提取到简报中，写作代理永远不需要重新阅读代码库。

### 阶段 3：构建课程

课程输出是一个**目录**，不是单个文件。所有 CSS 和 JS 都是预构建的参考文件——永不重新生成它们。你的工作是只编写 HTML 内容。

**输出结构：**
```
course-name/
  styles.css       ← 从 references/styles.css 原样复制
  main.js          ← 从 references/main.js 原样复制
  _base.html       ← 自定义外壳（标题、强调色、导航点）
  _footer.html     ← 从 references/_footer.html 原样复制
  build.sh         ← 从 references/build.sh 原样复制
  briefs/          ← 模块简报（仅限复杂代码库，构建后可删除）
  modules/
    01-intro.html
    02-actors.html
    ...
  index.html       ← 由 build.sh 组装（不要手动编写）
```

**步骤 1（两种路径）：设置** — 创建课程目录。使用读取 + 写入原样复制这四个文件（不要重新生成它们的内容）：
- `references/styles.css` → `course-name/styles.css`
- `references/main.js` → `course-name/main.js`
- `references/_footer.html` → `course-name/_footer.html`
- `references/build.sh` → `course-name/build.sh`

**步骤 2（两种路径）：自定义 `_base.html`** — 读取 `references/_base.html`，然后将其写入 `course-name/_base.html`，进行恰好三次替换：
- `COURSE_TITLE` 的两处 → 实际课程标题
- 四个 `ACCENT_*` 占位符 → 选择的强调色值（从 `_base.html` 的注释中选择一个调色板）
- `NAV_DOTS` → 每个模块一个 `<button class="nav-dot" ...>`

**步骤 3：编写模块** — 这里路径分开。

#### 顺序路径（简单代码库）

读取 `references/content-philosophy.md` 和 `references/gotchas.md`。然后一次编写一个模块。对于每个模块，编写 `course-name/modules/0N-slug.html`，只包含 `<section class="module" id="module-N">` 块及其内容。不要包含 `<html>`、`<head>`、`<body>`、`<style>` 或 `<script>` 标签。

阅读 `references/interactive-elements.md` 了解每种互动元素的 HTML 模式。阅读 `references/design-system.md` 了解视觉约定。

#### 并行路径（复杂代码库）

以最多 3 个一批的方式将模块分派给子代理。每个代理收到：
- 它的模块简报（来自 `course-name/briefs/`）
- `references/content-philosophy.md` 和 `references/gotchas.md`
- `references/interactive-elements.md` 和 `references/design-system.md` 中简报列出的唯一部分

每个代理将其模块文件写入 `course-name/modules/`。短模块（3 个屏幕，一个测验）可以配对——两个简报交给一个代理。

**代理不收到什么：** 完整代码库（片段在简报中）、SKILL.md、其他模块的简报或不需要的参考文件部分。

所有代理完成后，在主上下文中进行快速一致性检查：导航点与模块匹配、模块之间的过渡连贯、没有明显的语气变化。

**步骤 4（两种路径）：组装** — 从课程目录运行 `build.sh`：
```bash
cd course-name && bash build.sh
```
这会生成 `index.html`。在浏览器中打开它。

**关键规则：**
- **永不重新生成** `styles.css` 或 `main.js` — 始终从参考文件复制
- 模块文件只包含 `<section>` 内容——没有样板文件
- 使用 CSS `scroll-snap-type: y proximity`（不是 `mandatory`）
- 在 `.module` 上使用 `min-height: 100dvh` 和 `100vh` 后备
- 互动元素 JS 在 `main.js` 中；通过 `data-*` 属性和 CSS 类名连接，如 `references/interactive-elements.md` 中所示
- 聊天容器需要 `id` 属性；流动画需要在 `.flow-animation` 上使用 `data-steps='[...]'` JSON

### 阶段 4：审阅和打开

运行 `build.sh` 后，在浏览器中打开 `index.html`。引导用户了解构建的内容，并询问对内容、设计和互动性的反馈。

## 设计标识

视觉设计应该感觉像一个**精美的开发者笔记本**——温暖、引人入胜、与众不同。阅读 `references/design-system.md` 了解完整的令牌系统，但这里有一些不可妥协的原则：

- **温暖色调**：米白色背景（如陈年纸）、暖灰色、不能有冷白色或蓝色
- **大胆的强调色**：一种自信的强调色（朱红、珊瑚、青色——不是紫色渐变）
- **独特的排版**：有个性的展示字体用于标题（Bricolage Grotesque，或类似的粗几何字体——绝不能用 Inter、Roboto、Arial 或 Space Grotesk）。干净的无衬线字体用于正文（DM Sans 或类似）。JetBrains Mono 用于代码。
- **慷慨的留白**：模块有呼吸空间。每个屏幕最多 3-4 段短文。
- **交替背景**：偶数/奇数模块在两种暖背景色调之间交替，形成视觉节奏
- **深色代码块**：IDE 风格，深靛蓝炭色背景（#1E1E2E）上的 Catppuccin 风格语法高亮
- **深度但不刺眼**：微妙的暖色调阴影，绝不要纯黑色投影

## 参考文件

`references/` 目录包含详细规范。**只在到达相关阶段时才阅读它们**——不是提前。这可以保持上下文精简。

- **`references/content-philosophy.md`** — 视觉密度规则、隐喻指南、测验设计、提示规则、代码翻译指导。在阶段 2.5（简报）和阶段 3（编写模块）期间阅读。
- **`references/gotchas.md`** — 常见失败点检查清单。在阶段 3 和阶段 4（审阅）期间阅读。
- **`references/module-brief-template.md`** — 阶段 2.5 模块简报的模板。仅用于使用并行路径的复杂代码库。
- **`references/design-system.md`** — 完整的 CSS 自定义属性、调色板、排版比例、间距系统、阴影、动画、滚动条样式。在阶段 3 编写模块 HTML 时阅读。
- **`references/interactive-elements.md`** — 每种互动元素的实现模式：拖放匹配测验、选择题、代码↔英文对照、群聊动画、消息流可视化、架构图、模式卡、提示框。在阶段 3 阅读相关部分。
